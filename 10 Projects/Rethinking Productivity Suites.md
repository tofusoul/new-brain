# Rethinking Productivity Suites

**Status:** Concept Phase
**Type:** Philosophical/Design Project

## Core Problem

### The Paper Paradigm
Documents are presentations of information in human-readable form. Humans care about being able to make sense of information because what we do depends on that understanding.

**Why Paper Works:**
- Present information in a familiar way
- Helps you understand it
- Store and retrieve in physical space (something humans understand well)
- When they form a collection, they tell a narrative history
- That narrative history helps with human understanding

### The Computer Reality
Computers need well-structured data to reliably process them. For us to make good use of computers, we need to structure data well.

**Paper's Limitations:**
- Does not lead to well-structured data
- You can scribble in margins (unstructured)
- You can sketch instead of write
- You can change format to suit captured information
- All of this breaks structured data

### The Current Problem
While pieces of paper barely exist in people's workflows anymore, we still represent information as if they are on pieces of paper.

## The Proposed Solution

### Capture What's Good About Paper
- Familiar presentation
- Physical space organization
- Narrative history of collections

### Build Around Data, Not Documents
**Idea:** Office suite built around collectively creating data structures and processes to generate documents, rather than using documents to generate processes.

**Key Shift:**
- Current: Documents → Processes
- Proposed: Data Structures + Processes → Documents

## Core Thesis
Productivity suites need to be built on database management services, not file systems.

Web-based productivity suites like Google Docs are built on databases but emulate the experience of individuated files on file systems. We should embrace the database paradigm instead.
